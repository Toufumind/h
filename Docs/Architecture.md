# SpriteAssemble / Gemini Bond Architecture

## 1. Project Positioning

This project is a 2D pixel-art roguelike, chapter-map-based, wave-combat game.

The core experience is not just shooting. The core experience is:

```text
IceMirror auto-casts spells
Gem modules are arranged from left to right
The ordered gem chain behaves like a spell program
The player wins by composing and refining that spell program
```

Architecture target:

```text
Low coupling
High composition
Data-driven behavior
Event-driven presentation
Batch-managed shots
```

Core architectural style:

```text
Actor + Component + Spec + Context + Command + Subsystem + Event
```

Meaning:

```text
Actor      = identity, transform, lifecycle, component assembly
Component  = reusable gameplay capability
Spec       = behavior data passed between systems
Context    = current execution state of a spell chain
Command    = explicit cross-system mutation request
Subsystem  = global/batched runtime system
Event      = result notification
DataAsset  = authoring/tuning data
Blueprint  = asset binding, tuning, UI, level assembly
```

Core rules:

```text
Actor assembles capabilities; it does not own all logic.
Gem modifies data; it does not directly control actors.
Gem outputs Commands for cross-system effects instead of calling concrete systems.
Shot is data; it is not one Actor per projectile.
UI and Presentation listen to events; they do not run gameplay logic.
PaperZD, Flipbook, VFX, and SFX belong to Presentation.
Combat does not know UI, animation, or concrete player/enemy classes.
Enemy does not depend on the concrete PlayerCharacter class.
Cross-module writes go through Commands or Interfaces.
Cross-module reads go through Queries, Interfaces, or immutable state snapshots.
```

## 3. Architecture Decision Status

This document is an architecture contract, but not every item has the same decision weight.

### Locked Decisions

These are fixed architecture decisions:

```text
RunFlow is URunFlowWorldSubsystem.
Shot is not an Actor.
Shot runtime data is owned by UShotWorldSubsystem.
Current Shot rendering uses Instanced Mesh.
Gem depends on Core contracts only.
Gem modifies Specs and emits Commands.
Gem does not directly call Health, Effect, AreaEffect, Attribute, UI, or Presentation.
SpellCircuit owns ordered left-to-right gem execution.
New objects generated by a gem continue from NextGemIndex.
CommandDispatcher routes only; it does not own gameplay rules.
UI reads ViewModels/snapshots and emits commands.
Presentation reads events/state and does not run gameplay logic.
RunFlow orchestrates ChapterMap, Encounter, Reward, and Preparation.
ChapterMap, Encounter, Reward, and Preparation do not directly control each other.
RunState is the cross-node source of truth.
Scene components can be runtime projections of RunState.
Gameplay modules do not load heavy presentation assets directly.
Actor classes are composition roots, not gameplay rule owners.
```

### Implementation Contracts

These are required contracts, but exact class names can still be adjusted during implementation:

```text
FAttackSpec
FSpellExecutionContext
FShotSpawnSpec
FDamageSpec
FShotHitSpec
FGameplayCommand
FGameplayEvent

UGemRuntimeInstance

```

The responsibility boundaries are fixed. The final C++ names may change only if the boundary stays intact.

### Still Flexible

These are intentionally flexible:

```text
Exact fields inside each Spec.
Exact list of ViewModel classes.
Exact DataAsset class names.
Exact enemy action data schema.
Whether debug AreaEffects use temporary Actors internally.
Whether invulnerability is its own component or an Effect.
Future ShotRendererComponent internals after Instanced Mesh.
Exact target selection policies.
Exact reward weighting formulas.
```

Flexible does not mean unbounded. Any implementation must still obey dependency direction, state ownership, command dispatch, and event timing rules.

### Current Documentation Risks

The remaining risk is not module direction. The main risk is implementation drift:

```text
Putting gem rules back into PlayerCharacter.
Letting CommandDispatcher execute gameplay rules.
Letting RunFlow become a content/rule owner instead of an orchestrator.
Letting RunState and runtime components both mutate the same value.
Letting UI widgets call gameplay components directly.
Letting DataAssets hard-reference heavy presentation assets from gameplay modules.
Letting Shot systems expose array indices instead of FShotId.
Letting events trigger immediate gameplay mutation instead of queued Commands.
```

These risks should be checked during implementation reviews.

```text

```

Gem must not see:

```text

```

UI and Presentation are read/event consumers:

```text
UI           -> Core contracts, ViewModels, Run snapshots

```

Gameplay modules must not depend on UI or Presentation.

Concrete allowed dependencies:

```text
Attribute   -> Core
Combat      -> Core, Attribute
Effect      -> Core, Attribute, Combat
Gem         -> Core only
Weapon      -> Core, Combat, Shot

AreaEffect  -> Core, Combat, Effect

Encounter   -> Core, Enemy
Reward      -> Core, Command dispatch, Run snapshots
Economy     -> Core, Run snapshots

RunFlow     -> Core, Run, ChapterMap, Encounter, Reward, Preparation

```

Allowed dependency means:

```text
The module may use public contracts, interfaces, DataAsset types, and component attachment points.
The module may not reach into another module's internal mutable state.
The module may not bypass Commands/Queries for cross-module writes.
```

### Composition Root Exception

Actors and level setup sometimes need to assemble components from several modules.

This is allowed as composition, not logic ownership.

Examples:

```text
PlayerCharacter may own UGemInventoryComponent as an attached component.
EnemyActor may own UCombatComponent and UEnemyBrainComponent.
GameMode/Level may create Encounter and Run services.
```

Rules:

```text
Composition roots may create/attach components.
Composition roots may expose component references for editor binding.
Composition roots must not implement the owned component's business rules.
Composition roots must communicate through commands, queries, and events after setup.
```

This keeps Unreal-friendly assembly without turning PlayerCharacter, EnemyActor, or GameMode into god classes.

Forbidden dependencies:

```text
Shot -> PlayerCharacter
Shot -> Enemy concrete classes
Shot -> GemInventory
Gem -> PlayerCharacter
Gem -> FShotInstance
Combat -> UI
Combat -> Animation/PaperZD
HealthComponent -> Widget
Enemy -> PlayerCharacter
IceMirror -> UI
Presentation -> gameplay decisions
UI -> gameplay mutation, except explicit user commands such as buying or selecting rewards
Gem -> UHealthComponent
Gem -> UAttributeComponent

Reward -> GemInventory direct mutation

```

## 5. Core Contracts

Low coupling depends on strict shared contracts.

Core owns:

```text
Specs
Execution contexts
Commands
Events
Queries
Interfaces
Gameplay tags
Stable IDs
```

### Specs

Specs are behavior data. They can be created, modified, and consumed.

Examples:

```text
FAttackSpec
FShotSpawnSpec
FDamageSpec
FAreaEffectSpec
FStatusEffectSpec
FRewardSpec
```

### Contexts

Contexts carry execution state.

Examples:

```text
FSpellExecutionContext
FRunContext
FEncounterContext
```

### Commands

Commands are explicit mutation requests. They are produced by high-level systems and processed by the owning system.

Examples:

```text
FGameplayCommand_Heal
FGameplayCommand_ApplyStatusEffect
FGameplayCommand_CreateAreaEffect
FGameplayCommand_AddAttributeModifier
FGameplayCommand_AddGem
FGameplayCommand_UpgradeGem
FGameplayCommand_EquipGem
FGameplayCommand_ReorderGems
FGameplayCommand_AddRedCore
FGameplayCommand_SpendRedCore
FGameplayCommand_AddGemSlot
FGameplayCommand_AddPotionMax
FGameplayCommand_ReviveOnce
FGameplayCommand_StartEncounter
FGameplayCommand_CompleteNode
```

Command rules:

```text
The producer does not execute the command.
The owning module validates and executes the command.
Commands are auditable and testable.
Commands can fail with a reason.
Commands are better than direct cross-module mutation.
```

Example:

```text
LifestealGemEffect
-> emits FGameplayCommand_Heal(SourceActor, Amount)
-> Combat/Health command processor validates target
-> HealthComponent heals
-> OnHealed event is broadcast
```

### Command Dispatch

Command dispatch must not become a god object.

Use a small dispatcher that only routes commands:

```text
FGameplayCommand
-> CommandDispatcher.Route()
-> Owning module command processor
-> Validate
-> Execute
-> Emit event/result
```

The dispatcher may know command categories, but it must not own gameplay rules.

Command ownership:

```text
Heal / Damage / Revive
-> Combat / Health processor

ApplyStatusEffect / RemoveStatusEffect
-> Effect processor

CreateAreaEffect
-> AreaEffect processor

AddAttributeModifier / RemoveAttributeModifier
-> Attribute processor

AddGem / UpgradeGem / EquipGem / ReorderGems
-> Gem inventory processor

AddRedCore / SpendRedCore
-> Economy / Run processor

AddGemSlot / AddPotionMax / RefillPotions
-> Run or owning player-state processor

StartEncounter / CompleteEncounter
-> Encounter processor

CompleteNode / SelectNode
-> ChapterMap / Run processor
```

Command execution rules:

```text
Commands generated during spec mutation are queued.
Commands are executed at explicit resolution points.
Command processors can reject invalid commands.
Commands return a result object or emit failure events.
Commands should be deterministic for the same input state.
```

Recommended combat timing:

```text
Build specs
-> Gem modifies specs
-> Submit shots
-> Shot hit creates hit spec
-> SpellCircuit handles hit and appends commands
-> Combat resolves damage
-> Command queue resolves effects/heal/area effects
-> Events are broadcast
```

This prevents re-entrant chains where an event immediately mutates state while the original hit is still resolving.

### Events

Events report completed outcomes.

Examples:

```text
FGameplayEvent_AttackStarted
FGameplayEvent_ShotSpawned
FGameplayEvent_ShotHit
FGameplayEvent_Damaged
FGameplayEvent_Death
FGameplayEvent_GemTriggered
FGameplayEvent_StatusApplied
FGameplayEvent_AreaEffectCreated
FGameplayEvent_RunCurrencyChanged
FGameplayEvent_GemInventoryChanged
FGameplayEvent_NodeCompleted
```

Event rules:

```text
Events are facts, not requests.
Listeners should not mutate unrelated systems directly.
Presentation and UI consume events.
Gem may respond to gameplay events through SpellCircuit with context.
```

### Event Bus

Use a lightweight event bus or typed delegates for event distribution.

Rules:

```text
Event producers do not know listeners.
Event listeners subscribe by event type or gameplay tag.
Events are immutable after broadcast.
Events should carry ids, tags, actors, and values, not component references.
Events should be buffered per phase when deterministic order matters.
```

Recommended event phases:

```text
GameplayResolveEvents
PresentationEvents
UIEvents
DebugEvents
```

Presentation and UI should generally consume events after gameplay state has finished resolving for the current phase.

### Interfaces

Use interfaces for actor capability checks.

Examples:

```text
IDamageable
ITargetable
ITeamAgent
IAttributeSource
ICommandReceiver
IPickupReceiver
IPresentationStateProvider
```

Specs should prefer:

```text
AActor* or stable object id
TeamId
Gameplay tags
Numeric values
Context ids
```

Specs should avoid:

```text
UHealthComponent*
UGemInventoryComponent*
UAttributeComponent*
Concrete player/enemy classes
```

## 7. State Ownership Matrix

Every piece of state must have one owner.

```text
Current health
-> UHealthComponent

Base/final attributes
-> UAttributeComponent

Temporary buffs/debuffs/marks
-> UEffectComponent

Equipped gem order
-> RunState is source of truth between nodes
-> UGemInventoryComponent is runtime projection during combat/preparation scenes

Gem backpack
-> RunState is source of truth between nodes
-> UGemInventoryComponent is runtime projection during scenes

Gem definitions
-> UGemDefinition DataAssets

Current running shot data
-> UShotWorldSubsystem

Shot visual instances
-> AShotInstancedMeshView

Current combat encounter state
-> EncounterManager / WaveDirector

Current roguelike run state
-> RunState

Current flow step
-> RunFlow

Chapter node graph and node completion
-> ChapterMap

Currency
-> RunState owns current amount
-> Economy owns validation and price rules

Preparation shop stock
-> Preparation node state

Reward options
-> RewardManager until claimed

UI display state
-> ViewModels / immutable snapshots

Animation/VFX state
-> Presentation components
```

Rules:

```text
If two systems need the same state, one owns it and the other reads a snapshot/query.
If a system wants to change state it does not own, it sends a command.
If a system wants to react to a completed change, it listens to an event.
```

### Source of Truth vs Runtime Projection

Some systems need scene-local projections of run state.

Examples:

```text
RunState owns the saved/current run gem inventory.
UGemInventoryComponent is a scene runtime projection used by SpellCircuit and UI.

RunState owns current RedCore amount.
Economy validates prices and emits spend/add commands.

RunState owns potion count between nodes.
UPotionComponent is the runtime projection on the player during combat.
```

Rules:

```text
Only one side is the source of truth for a given scope.
Projection components are hydrated from RunState when entering a scene.
Projection changes are written back through commands or explicit scene-exit synchronization.
Do not let RunState and runtime components independently mutate the same value.
```

## 10. Runtime Phase Order

The runtime should use explicit phases so systems do not depend on accidental Tick order.

Recommended combat frame order:

```text
1. Input phase
   - Player input becomes commands.

2. AI phase
   - EnemyBrain chooses intent.
   - EnemyAction advances telegraph/startup/recovery timers.

3. Auto-cast phase
   - IceMirror timers tick.
   - Targeting queries targets.
   - SpellCircuit builds specs.

4. Shot movement phase
   - ShotWorldSubsystem updates FShotInstance positions.
   - Sweep queries generate FShotHitSpec.

5. Combat resolve phase
   - Combat resolves hit specs.
   - Health applies damage/heal.
   - Death and kill events are produced.

6. Command resolve phase
   - Queued commands apply effects, area effects, heals, attribute changes.
   - New commands generated here are queued for the next allowed resolution point unless explicitly marked immediate.

7. Encounter phase
   - WaveDirector checks deaths and wave completion.
   - Encounter victory/defeat is resolved.

8. Event flush phase
   - Gameplay events are finalized.
   - UI and Presentation receive event batches.

9. Render sync phase
   - ShotInstancedMeshView updates instance transforms.
   - Presentation components update visuals.
```

Rules:

```text
Do not rely on arbitrary Actor Tick order for gameplay correctness.
Subsystems that need deterministic order should be driven by an explicit coordinator.
Events that create gameplay consequences should go through SpellCircuit/Command queues, not immediate listener side effects.
```

## 11. Shot Instance Lifecycle

Shot runtime data must remain stable even when instances are removed.

Use:

```text
FShotId
FShotInstance
InstanceIndex
Generation
FreeList or swap-remove mapping
```

Rules:

```text
External systems reference shots by FShotId, not array index.
Only UShotWorldSubsystem can create or destroy FShotInstance.
AShotInstancedMeshView owns render instance indices.
ShotWorldSubsystem owns the mapping from FShotId to render instance handle.
Removing a shot must update mappings atomically.
Dead shots are marked during iteration and removed after iteration.
```

Recommended removal flow:

```text
Mark shot dead
-> Queue visual removal/update
-> Finish shot iteration
-> Compact shot array or free slot
-> Update FShotId mapping
-> Update InstancedMesh instance mapping
```

This avoids invalid indices during hit resolution, pierce handling, and event dispatch.

DataAssets can accidentally couple modules through hard references.

Use hard references for:

```text

Always-loaded combat definitions
```

Use soft references for:

```text
Large textures
VFX
SFX
Animation assets
Level assets
```


## 17. SpellCircuit Module

`SpellCircuit` is the architecture expression of "magic programming".

It executes equipped gems from left to right.

Core rule:

```text
The spell chain flows from left to right.
New things created by a gem continue only from the next gem index.
They never go back to earlier gems.
```

Example:

```text
Slot 1: Split
Slot 2: Pierce
Slot 3: Poison

Split creates secondary shots.
Those secondary shots start at NextGemIndex = 2.
They can receive Pierce and Poison.
They cannot trigger Split again.
```

Typical class:

```text
USpellCircuitComponent
```

Typical context:

```text
FSpellExecutionContext
- EquippedGemRuntimeIds
- CurrentGemIndex
- NextGemIndex
- ChainDepth
- TriggerType
- SourceActor
- InstigatorActor
- SourceShotId
- EventId
- bAllowSplit
- bAllowRecursiveGeneration
- MaxGeneratedShotCount
- MaxChainDepth
```

Trigger types:

```text
PreCast
OnBuildShot
OnBuildDamage
OnShotHit
OnKill
OnDamaged
OnTimer
```

SpellCircuit responsibilities:

```text
Execute gems in order
Maintain NextGemIndex
Prevent infinite chains
Route events to valid later gems
Expose deterministic execution rules
```

SpellCircuit should not:

```text
Own gem inventory storage
Own UI editing
Own shot movement
Own damage application
```


Current gem mappings:

```text

-> modifies FShotSpawnSpec
-> creates 3 sequential shots or burst requests
-> each shot may carry damage multiplier per level

Pierce
-> modifies FShotSpawnSpec / FDamageSpec shot runtime config fields
-> adds PierceCount
-> applies post-pierce damage multiplier

Split
-> modifies FShotSpawnSpec
-> creates secondary shots
-> secondary shots start at NextGemIndex = current index + 1
-> secondary shots cannot split again unless explicitly allowed

Swift
-> emits FGameplayCommand_AddAttributeModifier or modifies FAttackSpec
-> range and attack speed

Poison
-> handles FShotHitSpec
-> emits FGameplayCommand_ApplyStatusEffect

Counter
-> handles OnDamaged
-> emits FGameplayCommand_AddAttributeModifier or stores charges in Effect-owned state

Resonance
-> handles FShotHitSpec
-> emits FGameplayCommand_ApplyStatusEffect for mark stack
-> full stacks emit FGameplayCommand_CreateAreaEffect

FrostCrystal
-> handles FShotHitSpec
-> emits FGameplayCommand_CreateAreaEffect at hit location
```

## 30. RunFlow Module

RunFlow is implemented as:

```text
URunFlowWorldSubsystem
```

It is a thin orchestration layer.

It prevents ChapterMap, Encounter, Reward, and Preparation from directly controlling each other.

Responsibilities:

```text
Listen to node selection events
Start the correct node activity
Listen to activity completion events
Dispatch reward/preparation/map completion commands
Move the run to the next phase
```

RunFlow is not:

```text
An Actor placed in every map
A Component attached to PlayerCharacter
A UI controller
A reward generator
A wave spawner
A shop/economy rule owner
```

Reason:

```text
The flow belongs to the current World/run session.
It should not depend on a scene-placed manager actor.
It should not belong to any single gameplay actor.
It should survive local flow transitions inside the same world.
It should stay isolated from concrete UI widgets.
```

RunFlow states:

```text
MapSelection
EnteringCombat
Combat
Reward
Preparation
EventNode
Boss
RunWon
RunLost
```

Rules:

```text
ChapterMap does not start Encounter directly.
Encounter does not open Reward directly.
Reward does not advance ChapterMap directly.
Preparation does not advance ChapterMap directly.
RunFlow listens to their events and dispatches the next command.
```

Flow example:

```text
ChapterMap emits NodeSelected
-> RunFlow dispatches StartEncounter or OpenPreparation
-> Encounter emits EncounterCompleted
-> RunFlow dispatches BuildRewardOptions
-> Reward emits RewardClaimed
-> RunFlow dispatches CompleteNode
-> ChapterMap updates available next nodes
```

Runtime configuration flow:

```text
DefaultGame.ini or a composition root assigns URunConfigDefinition
-> URunConfigWorldSubsystem owns the active run config for the World
-> URewardManagerWorldSubsystem::ResolveRewardPool asks RunConfig for the RewardPool config object
-> RewardManager consumes URewardPoolDefinition to build reward options
-> RewardManager falls back only when no run config reward pool exists
```

Validation data asset:

```text
/Game/Data/Run/DA_RunConfig_Validation
-> ConfigObjects[RewardPool] = /Game/Data/Rewards/DA_RewardPool_Validation
```

Rules:

```text
RunConfig is the runtime entry point for run-scoped tuning data.
Reward does not depend on validation actors or level-only smoke setup.
Validation actors may still override data for smoke/bootstrap compatibility, but they are not the production configuration path.
RunConfig stores generic keyed config objects so Run does not need reward-generation rules.
Reward may cast only the RewardPool entry it owns and must not mutate RunConfig.
```

## 37. Main Runtime Data Flow

### Combat Shot Flow

```text
IceMirror timer
-> TargetingComponent.QueryTarget()
-> CombatComponent.BuildAttackSpec()
-> SpellCircuit creates FSpellExecutionContext
-> Gem chain modifies FAttackSpec
-> ShooterComponent.BuildShotSpawnSpec()
-> Gem chain modifies FShotSpawnSpec
-> CombatComponent.BuildDamageSpec()
-> Gem chain modifies FDamageSpec
-> ShotSpawnSpec carries DamageSpec and NextGemIndex
-> UShotWorldSubsystem.SubmitShot()
-> FShotInstance[]
-> AShotInstancedMeshView displays instances
-> Shot sweep hit detection
-> FShotHitSpec
-> SpellCircuit continues from NextGemIndex for hit gems
-> Gem effects append gameplay commands
-> CombatComponent.ResolveHit()
-> HealthComponent.ApplyDamage()
-> Command processors apply Effect / AreaEffect / Heal / Attribute changes
-> Events to UI and Presentation
```

### Encounter Flow

```text
ChapterMap selects combat node
-> EncounterConfig loaded
-> WaveDirector starts wave
-> SpawnDirector spawns enemies
-> EnemyBrain drives enemy actions
-> IceMirror and enemy attacks resolve combat
-> WaveDirector detects all enemies cleared
-> Next wave or encounter victory
-> RewardManager creates reward options
-> Player chooses reward
-> Reward command is dispatched
-> Owning module updates RunState/Gem/Attribute/Economy
-> RunFlow dispatches CompleteNode when appropriate
```

### Preparation Flow

```text
ChapterMap selects preparation node
-> Preparation scene/UI opens
-> Player interacts with merchant/craftsman/statue/workstation/mystery merchant
-> Economy validates cost
-> Command is dispatched
-> Owning module applies RunState / Gem / Attribute / Potion change
-> UI updates through events
-> Player exits preparation
-> RunFlow dispatches CompleteNode when appropriate
```

## 38. Composition Rules

Keep logic composable by following these rules:

```text
One component owns one gameplay capability.
One Spec carries one behavior snapshot.
One Context carries one execution-chain state.
One Subsystem owns one global/batched runtime collection.
One Event reports one completed outcome.
One Command requests one mutation from an owning module.
DataAssets define tuning; code defines execution contracts.
```

Prefer:

```text
GemEffect modifies FShotSpawnSpec.
GemEffect emits FGameplayCommand_ApplyStatusEffect.
Effect command processor applies Poison.
GemEffect emits FGameplayCommand_CreateAreaEffect.
AreaEffect command processor creates delayed explosion.
ShotWorldSubsystem moves shot data.
Presentation listens to FGameplayEvent_GemTriggered.
```

Avoid:

```text
PlayerCharacter calculates all gem behavior
Shot casts to PlayerCharacter
Enemy casts to PlayerCharacter
Gem directly changes FShotInstance
UI writes HealthComponent internals
Combat spawns VFX
Gem calls UHealthComponent directly
Reward directly changes GemInventory
Preparation directly changes AttributeComponent
```

## 39. Final Architecture Summary

The project should become:

```text
Run/ChapterMap controls the roguelike route.
Encounter controls one combat node.
IceMirror automatically drives the spell.
SpellCircuit executes ordered gems from left to right.
Gem modifies Specs and responds to events through Context.
Gem emits Commands for cross-system consequences.
Shot is lightweight data managed by UShotWorldSubsystem.
Instanced Mesh displays current shots.
Combat and Health consume damage data.
Effect and AreaEffect handle duration, stacks, marks, pull, poison, and explosions.
Reward/Economy/Preparation mutate RunState and components through explicit commands.
UI and Presentation listen to events.
```

This keeps systems low-coupled and highly composable while preserving the design goal:

```text
Build a spell like writing a small magic program.
```
